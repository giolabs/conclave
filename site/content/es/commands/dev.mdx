# /conclave-dev US-NNN [US-NNN ...]

Toma una o más historias de usuario del sprint activo y lleva cada una a través de la implementación. Cuando termina, cada historia queda en `status: review` con su propia feature branch y PR listo para verificación de QA.

```
/conclave-dev US-001                    # historia única — comportamiento idéntico al anterior
/conclave-dev US-001 US-002 US-003      # tres historias en paralelo, cada una su branch y PR
```

Se requiere al menos un argumento `US-NNN`; todos deben coincidir con archivos de historia bajo el sprint activo. Cuando se dan múltiples IDs, las historias se procesan en batches concurrentes de ≤ 3 y cada una obtiene una branch y un PR independientes. El camino de historia única es completamente retrocompatible.

## Comportamiento multi-historia

Cuando se pasa más de un ID de historia:

1. **Validación anticipada** — todas las historias se verifican antes de que empiece cualquier trabajo. Si alguna historia falla (no se encuentra, estado incorrecto, ya `in-progress` en otra branch), se rechaza toda la invocación con una tabla de errores por historia.
2. **Batches concurrentes** — las historias se particionan en grupos de ≤ 3. Todas las historias de un batch corren simultáneamente; el siguiente batch empieza recién cuando el anterior termina.
3. **Aislamiento de fallos** — una historia que falla se resetea a `status: ready`; el resto del batch continúa. Una tabla de resumen final muestra el resultado de cada historia.

```
| Historia | Branch                  | PR                           | Resultado         |
|----------|-------------------------|------------------------------|-------------------|
| US-001   | feat/US-001-login       | https://github.com/…/pull/42 | ✓ done            |
| US-002   | feat/US-002-profile     | https://github.com/…/pull/43 | ✓ done            |
| US-003   | feat/US-003-settings    | —                            | ✗ falló: <error>  |
```

## Qué hace

1. Verifica que el árbol de trabajo esté limpio (se rehúsa si está sucio).
2. Ubica el sprint activo y el archivo de la historia. Se rehúsa si la historia ya pasó el gate de dev.
3. Chequea el asignado — si no coincide con vos, pregunta si querés tomar la historia.
4. Carga contexto: `config.md` (perfil + flag de peer-review), arquitectura, DoD, roster (con un aviso de compatibilidad único si es anterior a la columna `Discipline`), historia, aceptación.
5. Crea la branch `feat/US-NNN-<slug>` desde la branch de integración.
6. Marca la historia `in-progress` en su propio commit (visible para el equipo de inmediato).
7. **Lee la `discipline` de la historia y delega al subagente que corresponda.** El agente:
   - Lee historia, aceptación, arquitectura, DoD.
   - Planea un desglose técnico (solo borrador — no se commitea).
   - Detecta o bootstrapea el setup de test/verificación.
   - Implementa historia-y-verifica, escenario por escenario. Cada escenario Gherkin obtiene al menos una forma de verificación (un test que pasa para Developer/DevOps, una decisión de diseño para Designer).
   - Corre la suite completa de tests / chequeo final una vez al terminar.
   - Corre linter / typecheck (cuando aplica).
   - Commitea en pedazos acotados (`feat(US-NNN): ...`).
   - Renderiza el cuerpo del PR desde `templates/pr-body.template.md`.
8. Pushea la branch.
9. Abre el PR vía `gh pr create` (o imprime el comando preparado si `gh` no está disponible).
10. Etiqueta a un peer reviewer si `peer_pr_review.required: true` — elige a alguien del roster, excluyendo al asignado.
11. Marca la historia `status: review`.

## Ruteo por disciplina

| `discipline` de la historia | Charter |
|---|---|
| `frontend`, `backend`, `multi`, o sin definir (historias pre-0.2.0) | `developer.md` |
| `design` | `designer.md` — solo artefactos de diseño y notas de handoff a Frontend, nunca código de aplicación |
| `devops` | `devops.md` — pipelines de CI/CD e infraestructura como código, el mismo loop de implementar → verificar → PR que Developer |

Los tres charters comparten los mismos inputs, la misma forma de outputs, y la misma plantilla de cuerpo de PR — solo difiere el dominio de lo que se construye.

## Qué produce

- Código (o, para `design`, notas de handoff) + evidencia de verificación en el repo.
- Una feature branch `feat/US-NNN-<slug>` pusheada a `origin`.
- Un PR (o el comando `gh pr create` preparado).
- Frontmatter de la historia actualizado: `assignee`, `status: review`.

## Reglas duras que sigue el subagente de ejecución

- Cada escenario Gherkin mapea a al menos una forma de verificación.
- Ninguna desviación arquitectónica sin una propuesta de ADR en el cuerpo del PR.
- Nunca modifica los criterios de aceptación. Si parecen incorrectos, lo marca vía un comentario, no lo arregla silenciosamente.
- No toca ningún archivo bajo `conclave/` excepto el frontmatter de su propio archivo de historia.
- Nunca mergea su propio PR.

## Consciencia del perfil

- `peer_pr_review.required: true` → etiqueta a un par (típicamente el Tech Lead) como reviewer, la checklist condicional del cuerpo del PR incluye el ítem de aprobación del TL.
- `peer_pr_review.required: false` → no se requiere un reviewer separado. El Dev igual se auto-revisa.

## Cambio de salvaguarda desde v0.5.0

Anteriormente el comando se rehusaba a arrancar si alguna historia ya estaba `in-progress`. La regla actualizada: una historia se rechaza solo si **ese `US-NNN` exacto** ya está `in-progress` en una branch existente. Que otras historias estén `in-progress` en sus propias branches está permitido y es esperado.

## Resumir en una branch existente

Si `feat/US-NNN-<slug>` ya existe localmente (la historia está `in-progress` de una corrida anterior), el comando ofrece tres opciones: cambiar a esa branch y resumir, borrar y recrear, o abortar. Al resumir, el subagente de ejecución lee lo que ya existe en la branch antes de agregar código nuevo.

## Modo autónomo (v0.9.0+)

`/conclave-dev` soporta un modo headless que nunca llama a `AskUserQuestion`. Se activa de tres formas:

```bash
# 1. Flag CLI ad-hoc (prioridad más alta; overridea config.md)
/conclave-dev --no-interaction US-042    # --headless es un sinónimo

# 2. Default del repo — agregar a conclave/config.md
commands:
  dev:
    interactive: false

# 3. Implícito dentro de /conclave-sprint — la Fase 2 fuerza autonomous mode independientemente de config.md
/conclave-sprint
```

### Qué cambia

- **Sin prompts.** Cada sitio con `AskUserQuestion` aplica un default documentado del catálogo de "sensible defaults". Los casos sin default seguro abortan con una razón específica.
- **Reporte por corrida appendeado al archivo de la historia** como una nueva sección `## Autonomous run — <ISO>` (mirá [`skills/conclave/templates/autonomous-run.template.md`](https://github.com/giolabs/conclave/blob/main/skills/conclave/templates/autonomous-run.template.md)).
- **Línea `Mode: autonomous`** printeada una vez al inicio de la corrida. El modo interactivo es silencioso — la ausencia de `Mode:` significa interactivo.

### Catálogo de sensible defaults

| Prompt interactivo | Default autónomo |
|---|---|
| Assignee mismatch (`assignee` de la historia ≠ `git config user.name` del runner) | **Take over** — reasigna al runner; se graba como `assignee_takeover: <old> → <new>` |
| Branch local existente sin commits de la historia | **Borrar y recrear** desde la branch de integración; se graba como `branch_recreated` |
| Branch local existente con commits previos de la historia hechos por el mismo runner | **Switch and resume**; se graba como `branch_resumed: from <sha>` |
| Branch local existente con commits de otro `git config user.email` | **Abort** — `AUTONOMOUS_ABORT: story branch has commits from another dev (<their email>); manual coordination required` |

### Escenarios de abort (lado del subagent)

El subagent Developer devuelve `AUTONOMOUS_ABORT: <reason>` cuando:

- **No se detecta test framework** — la historia requeriría bootstrapear uno (decisión humana).
- **Nueva dependencia requerida no listada en ningún ADR** — agregar una dep es un cambio de scope; el TL debería haberla autorizado vía `/conclave-adr` primero.
- **Escenario Gherkin ambiguo** — dos interpretaciones plausibles y nada (ADR, texto de la historia, notas técnicas) desambigua.
- **La historia requiere un cambio arquitectónico** — necesita `/conclave-adr` primero, no `/conclave-dev`.

En abort, el frontmatter de la historia se resetea a `status: ready`. No se pushea nada. Una sección `## Autonomous run —` con `Outcome: aborted` y la línea de abort bajo `Blockers` se appendea al archivo de la historia. Re-correr en modo interactivo (`/conclave-dev US-NNN` sin `--no-interaction`) para resolver la ambigüedad.

### Ejemplo de sección de run-report

```markdown
## Autonomous run — 2026-07-06T15:23:11Z

- **Outcome**: done
- **Branch**: `feat/US-042-jwt-middleware`
- **PR**: https://github.com/foo/bar/pull/123
- **Duration**: 4m 22s
- **Runner**: gio (git config user.email = gio@example.com)
- **Config source**: --no-interaction CLI flag

### Autonomous decisions

- assignee_takeover: alice → gio — story executed by whoever ran the batch
- test framework selection: vitest — already in package.json; architecture.md Confirmed stack lists it

### Files touched

- `src/middleware/jwt.ts` (new, 47 lines)
- `tests/middleware/jwt.test.ts` (new, 6 tests)

### Tests

- Gherkin scenarios covered: 3/3
- Suite: `pnpm test` → 342 pass, 0 fail
- Lint: 0 warnings
```

Para outcomes `blocked` y `aborted`, la sección adicionalmente incluye una subsección `### Blockers` con la línea de falla o abort verbatim.

### Asimetría: sin flag `--force-interaction`

Intencionalmente no hay un flag CLI para forzar modo interactivo cuando `config.md` dice autónomo. Un job de CI con `interactive: false` en su fork de `config.md` no debe poder colgarse por un flag suelto. Para debuggear, editá `config.md` localmente o usá otro repo/branch.
